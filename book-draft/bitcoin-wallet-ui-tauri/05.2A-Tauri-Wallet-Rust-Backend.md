# Chapter 19A: Tauri Wallet — Rust Backend (Complete Code Walkthrough)

This companion chapter contains the **complete, verbatim Rust backend source** for the `bitcoin-wallet-ui-tauri` application. Every file is reproduced in full with detailed annotations. We walk through the code in dependency order: configuration first, then database, then services, then commands, then the entry point that wires them together.

---

## Listing 23A.1: `src-tauri/Cargo.toml`


> - Package configuration and all dependency declarations

The wallet app's dependency list is shorter than the admin app's (Chapter 17A) but adds two critical entries: `rusqlite` with the `bundled-sqlcipher` feature for encrypted database support, and `dirs` for cross-platform data directory resolution.

```toml
[package]
name = "bitcoin-wallet-ui-tauri"
version = "0.1.0"
edition = "2021"

[lib]
name = "bitcoin_wallet_ui_tauri_lib"
crate-type = ["lib", "cdylib", "staticlib"]

[build-dependencies]
tauri-build = { version = "2", features = [] }

[dependencies]
tauri = { version = "2", features = [] }
tauri-plugin-clipboard-manager = "2"
tauri-plugin-shell = "2"
serde = { version = "1", features = ["derive"] }
serde_json = "1.0"
tokio = { version = "1", features = ["rt-multi-thread", "macros"] }
tracing = "0.1"
tracing-subscriber = "0.3"
dirs = "5.0"
rusqlite = { version = "0.31", features = ["bundled-sqlcipher"] }
bitcoin-api = { path = "../../bitcoin-api", features = ["client", "wallet", "admin"], default-features = false }
```

Key dependency notes:

- **`rusqlite` with `bundled-sqlcipher`**: This bundles SQLCipher (an encrypted fork of SQLite) directly into the binary. No external SQLCipher installation is needed. The `bundled` part compiles SQLCipher from source during `cargo build`.
- **`dirs = "5.0"`**: Provides `dirs::data_dir()` which resolves to `~/Library/Application Support` on macOS, `~/.local/share` on Linux, and `%APPDATA%` on Windows.
- **`bitcoin-api`**: Same shared crate used by all frontends, but here we enable the `wallet` feature (for `WalletClient`) in addition to `client` and `admin`.
- **`crate-type`**: The `["lib", "cdylib", "staticlib"]` triple is standard Tauri 2 — it supports both desktop (`cdylib`) and mobile (`staticlib`) targets, plus a regular `lib` for unit testing.

## Listing 23A.2: `src-tauri/src/config/mod.rs`


> - `AppState` struct and its `Default` and `api_config` implementations

The wallet's `AppState` differs from the admin app's `ApiConfig` in two important ways: it includes an `active_wallet` field for tracking which wallet is currently selected, and it loads its initial values from the database rather than environment variables.

```rust
use serde::{Serialize, Deserialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct AppState {
    pub active_wallet: Option<String>,
    pub base_url: String,
    pub api_key: String,
}

impl Default for AppState {
    fn default() -> Self {
        // Try to load settings from database
        let (base_url, api_key) = match crate::database::load_settings() {
            Ok(settings) => (settings.base_url, settings.api_key),
            Err(_) => (
                "http://127.0.0.1:8080".to_string(),
                std::env::var("BITCOIN_API_WALLET_KEY")
                    .unwrap_or_else(|_| "wallet-secret".to_string()),
            ),
        };
        Self {
            active_wallet: None,
            base_url,
            api_key,
        }
    }
}

impl AppState {
    pub fn api_config(&self) -> bitcoin_api::ApiConfig {
        bitcoin_api::ApiConfig {
            base_url: self.base_url.clone(),
            api_key: Some(self.api_key.clone()),
        }
    }
}
```

The `api_config()` method bridges the wallet's `AppState` into the `bitcoin-api` crate's `ApiConfig`. This lets commands write `state.read()?.api_config()` to get a fresh config for each API call.

The fallback chain in `Default::default()` is deliberate:
1. First, try loading from the encrypted database (user's saved preferences)
2. If that fails, try the `BITCOIN_API_WALLET_KEY` environment variable
3. If that also fails, use the hardcoded default `"wallet-secret"`

This means the app works immediately after first install (using defaults), and gets progressively more personalized as the user saves settings.

The wallet app has only two custom models because most data types come from the `bitcoin-api` crate (`ApiResponse`, `CreateWalletResponse`, etc.) and the database module (`WalletAddress`, `Settings`).

```rust
use serde::{Serialize, Deserialize};
```

`SendTxResult` is a thin wrapper that extracts just the transaction ID from the full `SendTransactionResponse`. This keeps the IPC payload minimal — the frontend only needs the txid to display a success message.

## Listing 23A.4: `src-tauri/src/services/bitcoin_api.rs`


> - `BitcoinApiService` with 6 async methods: `create_wallet`, `get_wallet_info`, `get_balance`, `send_transaction`, `get_address_transactions`, `health_check`

The service layer follows the same stateless pattern as the admin app (Chapter 17A): no fields, fresh client per call. But the wallet app uses `WalletClient` instead of `AdminClient` for most operations.

```rust
use bitcoin_api::{
    ApiConfig, ApiResponse, CreateWalletRequest, CreateWalletResponse,
    SendTransactionRequest, SendTransactionResponse, WalletClient,
};
use serde_json::Value;

pub struct BitcoinApiService;

impl BitcoinApiService {
    pub async fn create_wallet(
        cfg: ApiConfig,
        label: Option<String>,
    ) -> Result<ApiResponse<CreateWalletResponse>, String> {
        let client = WalletClient::new(cfg).map_err(|e| e.to_string())?;
        let req = CreateWalletRequest { label };
        client.create_wallet(&req).await.map_err(|e| e.to_string())
    }

    pub async fn get_wallet_info(
        cfg: ApiConfig,
        address: &str,
    ) -> Result<ApiResponse<Value>, String> {
        let client = WalletClient::new(cfg).map_err(|e| e.to_string())?;
        client.get_wallet_info(address).await.map_err(|e| e.to_string())
    }

    pub async fn get_balance(
        cfg: ApiConfig,
        address: &str,
    ) -> Result<ApiResponse<Value>, String> {
        let client = WalletClient::new(cfg).map_err(|e| e.to_string())?;
        client.get_balance(address).await.map_err(|e| e.to_string())
    }

    pub async fn send_transaction(
        cfg: ApiConfig,
        from: &str,
        to: &str,
        amount: u64,
    ) -> Result<ApiResponse<SendTransactionResponse>, String> {
        let client = WalletClient::new(cfg).map_err(|e| e.to_string())?;
        let req = SendTransactionRequest {
            from_address: from.to_string(),
            to_address: to.to_string(),
            amount,
        };
        client.send_transaction(&req).await.map_err(|e| e.to_string())
    }

    pub async fn get_address_transactions(
        cfg: ApiConfig,
        address: &str,
    ) -> Result<ApiResponse<Value>, String> {
        let client = WalletClient::new(cfg).map_err(|e| e.to_string())?;
        client.get_address_transactions(address).await.map_err(|e| e.to_string())
    }

    pub async fn health_check(cfg: ApiConfig) -> Result<bool, String> {
        let client = bitcoin_api::AdminClient::new(cfg).map_err(|e| e.to_string())?;
        match client.health().await {
            Ok(resp) => Ok(resp.success),
            Err(_) => Ok(false),
        }
    }
}
```

Notice that `health_check` uses `AdminClient` instead of `WalletClient`. The health endpoint is an admin-level concern, not a wallet-specific one. The wallet app includes it so the Settings page can verify that the connection is working.

The pattern of creating a fresh `WalletClient` per call (rather than holding a reference) means configuration changes take effect immediately — no restart required.

## Listing 23A.5: `src-tauri/src/database/mod.rs`


> - `init_database`, `get_connection`, `get_database_path`
> - `create_tables`, `run_migrations`
> - `WalletAddress`, `Settings` types
> - CRUD: `load_settings`, `save_settings_db`, `save_wallet_address`, `load_wallet_addresses`, `delete_wallet_address_db`, `update_wallet_label_db`

This is the most substantial module in the wallet app. It manages all local persistence through an encrypted SQLCipher database. The database module is **shared with the Iced wallet** (Chapter 18/6) — both apps use the same schema, the same key generation, and the same CRUD operations.

```rust
use rusqlite::{Connection, Result as SqliteResult, params};
use std::path::PathBuf;
use std::sync::Mutex;
use std::sync::OnceLock;
use serde::{Serialize, Deserialize};

type TableColumnInfo = (
    String, String, Option<String>, Option<String>, Option<String>, bool,
);

const DB_FILENAME: &str = "bitcoin-wallet.db";
static DB_CONN: OnceLock<Mutex<Connection>> = OnceLock::new();
const SCHEMA_VERSION: i32 = 2;
```

The `DB_CONN` global uses `OnceLock<Mutex<Connection>>` — a pattern that guarantees the connection is initialized exactly once and accessed safely from any thread. `OnceLock` was stabilized in Rust 1.70 and is the preferred replacement for `lazy_static!` when you need a one-time initialization.

### Initialization

```rust
pub fn init_database(password: &str) -> SqliteResult<()> {
    let db_path = get_database_path();

    if let Some(parent) = db_path.parent() {
        std::fs::create_dir_all(parent).map_err(|e| {
            rusqlite::Error::SqliteFailure(
                rusqlite::ffi::Error::new(rusqlite::ffi::SQLITE_CANTOPEN),
                Some(format!("Failed to create database directory: {}", e)),
            )
        })?;
    }

    let conn = Connection::open(&db_path)?;
    conn.pragma_update(None, "key", password)?;
    conn.pragma_update(None, "foreign_keys", "ON")?;
    create_tables(&conn)?;
    run_migrations(&conn)?;

    DB_CONN.set(Mutex::new(conn)).map_err(|_| {
        rusqlite::Error::SqliteFailure(
            rusqlite::ffi::Error::new(rusqlite::ffi::SQLITE_MISUSE),
            Some("Database already initialized".to_string()),
        )
    })?;

    Ok(())
}
```

The initialization sequence is critical:
1. **Create the directory** if it does not exist (`create_dir_all` is idempotent)
2. **Open the connection** — this creates the file if new, or opens the existing one
3. **Set the encryption key** via `PRAGMA key` — this must be the first statement after opening, or SQLCipher will treat the database as unencrypted
4. **Enable foreign keys** — SQLite has foreign keys off by default
5. **Create tables** — all `CREATE TABLE IF NOT EXISTS`, so safe to re-run
6. **Run migrations** — version-checked, idempotent
7. **Store the connection** in the global `OnceLock`

### Connection accessor

```rust
fn get_connection() -> SqliteResult<std::sync::MutexGuard<'static, Connection>> {
    DB_CONN
        .get()
        .ok_or_else(|| {
            rusqlite::Error::SqliteFailure(
                rusqlite::ffi::Error::new(rusqlite::ffi::SQLITE_MISUSE),
                Some("Database not initialized".to_string()),
            )
        })?
        .lock()
        .map_err(|_| {
            rusqlite::Error::SqliteFailure(
                rusqlite::ffi::Error::new(rusqlite::ffi::SQLITE_MISUSE),
                Some("Failed to acquire database lock".to_string()),
            )
        })
}
```

Every CRUD function calls `get_connection()` to obtain a `MutexGuard<Connection>`. The `Mutex` ensures that only one thread accesses the database at a time — SQLite (and SQLCipher) do not support concurrent writers on the same connection.

### Database path

```rust
fn get_database_path() -> PathBuf {
    let mut path = dirs::data_dir().unwrap_or_else(|| PathBuf::from("."));
    path.push("bitcoin-wallet-ui");
    path.push(DB_FILENAME);
    path
}
```

This resolves to the OS-specific application data directory. Using `dirs::data_dir()` is the correct choice for persistent application data — it survives app updates and is backed up by default on most platforms.

### Table creation

```rust
fn create_tables(conn: &Connection) -> SqliteResult<()> {
    conn.execute(
        "CREATE TABLE IF NOT EXISTS settings (
            id INTEGER PRIMARY KEY CHECK (id = 1),
            base_url TEXT NOT NULL DEFAULT 'http://127.0.0.1:8080',
            api_key TEXT NOT NULL DEFAULT '',
            created_at TEXT NOT NULL DEFAULT (datetime('now')),
            updated_at TEXT NOT NULL DEFAULT (datetime('now'))
        )",
        [],
    )?;

    let count: i64 = conn.query_row("SELECT COUNT(*) FROM settings", [], |row| row.get(0))?;
    if count == 0 {
        conn.execute(
            "INSERT INTO settings (id, base_url, api_key) VALUES (1, 'http://127.0.0.1:8080', ?)",
            params![
                std::env::var("BITCOIN_API_WALLET_KEY")
                    .unwrap_or_else(|_| "wallet-secret".to_string())
            ],
        )?;
    }

    conn.execute(
        "CREATE TABLE IF NOT EXISTS wallet_addresses (
            id INTEGER PRIMARY KEY AUTOINCREMENT,
            address TEXT NOT NULL UNIQUE,
            label TEXT,
            created_at TEXT NOT NULL DEFAULT (datetime('now')),
            updated_at TEXT NOT NULL DEFAULT (datetime('now'))
        )",
        [],
    )?;

    conn.execute(
        "CREATE INDEX IF NOT EXISTS idx_wallet_addresses_address ON wallet_addresses(address)",
        [],
    )?;

    conn.execute(
        "CREATE TABLE IF NOT EXISTS users (
            id INTEGER PRIMARY KEY CHECK (id = 1),
            first_name TEXT,
            last_name TEXT,
            profile_picture BLOB,
            created_at TEXT NOT NULL DEFAULT (datetime('now')),
            updated_at TEXT NOT NULL DEFAULT (datetime('now'))
        )",
        [],
    )?;

    conn.execute(
        "CREATE TABLE IF NOT EXISTS schema_version (
            version INTEGER PRIMARY KEY
        )",
        [],
    )?;

    let version_count: i64 =
        conn.query_row("SELECT COUNT(*) FROM schema_version", [], |row| row.get(0))?;
    if version_count == 0 {
        conn.execute(
            "INSERT INTO schema_version (version) VALUES (?)",
            params![SCHEMA_VERSION],
        )?;
    }

    Ok(())
}
```

Design decisions to notice:

- The `settings` table uses `CHECK (id = 1)` to enforce a singleton pattern — there can only ever be one settings row. This prevents accidental duplication while allowing standard SQL UPDATE operations.
- The `wallet_addresses` table has a `UNIQUE` constraint on `address`, which is enforced by SQLite at the database level. The `save_wallet_address` function below handles the duplicate case gracefully.
- Timestamps use `datetime('now')` — SQLite's built-in UTC datetime function.
- The `users` table is also a singleton (for profile data). The `profile_picture` column stores image data as a `BLOB` directly in the database (this was changed from a file path in schema v1 → v2).

### Migrations

```rust
fn run_migrations(conn: &Connection) -> SqliteResult<()> {
    let current_version: i32 = conn
        .query_row("SELECT version FROM schema_version LIMIT 1", [], |row| row.get(0))
        .unwrap_or(0);

    if current_version < 2 {
        let table_info: Vec<TableColumnInfo> = conn
            .prepare("PRAGMA table_info(users)")?
            .query_map([], |row| {
                Ok((
                    row.get::<_, String>(1)?,
                    row.get::<_, String>(2)?,
                    row.get(3)?,
                    row.get(4)?,
                    row.get(5)?,
                    false,
                ))
            })?
            .collect::<Result<Vec<_>, _>>()?;

        let has_old_column = table_info.iter().any(|(name, _, _, _, _, _)| name == "profile_picture_path");
        let has_new_column = table_info.iter().any(|(name, _, _, _, _, _)| name == "profile_picture");

        if has_old_column && !has_new_column {
            conn.execute_batch(
                "BEGIN TRANSACTION;
                 CREATE TABLE users_new (
                     id INTEGER PRIMARY KEY CHECK (id = 1),
                     first_name TEXT,
                     last_name TEXT,
                     profile_picture BLOB,
                     created_at TEXT NOT NULL DEFAULT (datetime('now')),
                     updated_at TEXT NOT NULL DEFAULT (datetime('now'))
                 );
                 INSERT INTO users_new (id, first_name, last_name, created_at, updated_at)
                 SELECT id, first_name, last_name, created_at, updated_at FROM users;
                 DROP TABLE users;
                 ALTER TABLE users_new RENAME TO users;
                 COMMIT;",
            )?;
        } else if !has_new_column {
            conn.execute("ALTER TABLE users ADD COLUMN profile_picture BLOB", [])?;
            if has_old_column {
                conn.execute_batch(
                    "BEGIN TRANSACTION;
                     CREATE TABLE users_new (
                         id INTEGER PRIMARY KEY CHECK (id = 1),
                         first_name TEXT,
                         last_name TEXT,
                         profile_picture BLOB,
                         created_at TEXT NOT NULL DEFAULT (datetime('now')),
                         updated_at TEXT NOT NULL DEFAULT (datetime('now'))
                     );
                     INSERT INTO users_new (id, first_name, last_name, created_at, updated_at)
                     SELECT id, first_name, last_name, created_at, updated_at FROM users;
                     DROP TABLE users;
                     ALTER TABLE users_new RENAME TO users;
                     COMMIT;",
                )?;
            }
        }

        conn.execute("UPDATE schema_version SET version = ?", params![2])?;
    }

    if current_version < SCHEMA_VERSION {
        conn.execute("UPDATE schema_version SET version = ?", params![SCHEMA_VERSION])?;
    }

    Ok(())
}
```

The migration is defensive: it uses `PRAGMA table_info(users)` to inspect the actual column layout before deciding what to do. This handles three possible states:
1. Old schema with `profile_picture_path` but no `profile_picture` → full table rebuild
2. No profile column at all → add the column, then rebuild if needed
3. Already migrated → skip

The "create new table, copy data, drop old, rename" pattern is the standard SQLite approach to column modifications — SQLite does not support `ALTER TABLE ... DROP COLUMN` in older versions.

### Public types

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct Settings {
```

`WalletAddress::new` sets `created_at` to an empty string because the database populates the timestamp via `DEFAULT (datetime('now'))`. After saving and re-reading, the `created_at` field will contain the actual timestamp.

### CRUD operations

```rust
pub fn load_settings() -> SqliteResult<Settings> {
    let conn = get_connection()?;
    conn.query_row(
        "SELECT base_url, api_key FROM settings WHERE id = 1",
        [],
        |row| Ok(Settings { base_url: row.get(0)?, api_key: row.get(1)? }),
    )
}

pub fn save_settings_db(settings: &Settings) -> SqliteResult<()> {
    let conn = get_connection()?;
    conn.execute(
        "UPDATE settings SET base_url = ?, api_key = ?, updated_at = datetime('now') WHERE id = 1",
        params![settings.base_url, settings.api_key],
    )?;
    Ok(())
}

pub fn save_wallet_address(wallet: &WalletAddress) -> SqliteResult<WalletAddress> {
    let conn = get_connection()?;
    match conn.execute(
        "INSERT INTO wallet_addresses (address, label, updated_at) VALUES (?, ?, datetime('now'))",
        params![wallet.address, wallet.label],
    ) {
        Ok(_) => {
            conn.query_row(
                "SELECT address, label, created_at FROM wallet_addresses WHERE address = ?",
                params![wallet.address],
                |row| Ok(WalletAddress { address: row.get(0)?, label: row.get(1)?, created_at: row.get(2)? }),
            )
        }
        Err(rusqlite::Error::SqliteFailure(err, _))
            if err.code == rusqlite::ErrorCode::ConstraintViolation =>
        {
            conn.execute(
                "UPDATE wallet_addresses SET label = ?, updated_at = datetime('now') WHERE address = ?",
                params![wallet.label, wallet.address],
            )?;
            conn.query_row(
                "SELECT address, label, created_at FROM wallet_addresses WHERE address = ?",
                params![wallet.address],
                |row| Ok(WalletAddress { address: row.get(0)?, label: row.get(1)?, created_at: row.get(2)? }),
            )
        }
        Err(e) => Err(e),
    }
}

pub fn load_wallet_addresses() -> SqliteResult<Vec<WalletAddress>> {
    let conn = get_connection()?;
    let mut stmt = conn.prepare(
        "SELECT address, label, created_at FROM wallet_addresses ORDER BY created_at DESC",
    )?;
    let addresses = stmt.query_map([], |row| {
        Ok(WalletAddress { address: row.get(0)?, label: row.get(1)?, created_at: row.get(2)? })
    })?;
    let mut result = Vec::new();
    for address in addresses {
        result.push(address?);
    }
    Ok(result)
}

pub fn delete_wallet_address_db(address: &str) -> SqliteResult<()> {
    let conn = get_connection()?;
    conn.execute("DELETE FROM wallet_addresses WHERE address = ?", params![address])?;
    Ok(())
}

pub fn update_wallet_label_db(address: &str, label: &str) -> SqliteResult<()> {
    let conn = get_connection()?;
    conn.execute(
        "UPDATE wallet_addresses SET label = ?, updated_at = datetime('now') WHERE address = ?",
        params![label, address],
    )?;
    Ok(())
}
```

The `save_wallet_address` function implements an **upsert pattern**: it tries to INSERT first, and if the address already exists (UNIQUE constraint violation), it falls back to UPDATE. In both cases, it returns the full `WalletAddress` with the database-populated `created_at` timestamp.

The `load_wallet_addresses` function orders by `created_at DESC` — most recently created wallets appear first.

The wallet commands fall into three categories:

**Local-only commands** (database access, no network):
- `get_saved_wallets` — reads from SQLCipher
- `set_active_wallet` — validates against DB, then updates in-memory state
- `delete_wallet_address` — removes from DB, clears active if needed
- `update_wallet_label` — updates label in DB

**Remote-only commands** (network, no database):
- `get_wallet_info`, `get_balance`, `get_tx_history` — fetch from Bitcoin node API

**Two-phase commands** (network + database):
- `create_wallet` — creates on node, persists locally
- `send_transaction` — sends via node (no local persistence)

Notice that `set_active_wallet` **validates** the address against the database before accepting it. This prevents the frontend from setting an active wallet that does not exist in the local store.

Also notice that `delete_wallet_address` checks if the deleted address is the currently active one, and clears it if so. This prevents a stale `active_wallet` reference pointing to a deleted wallet.

```rust
/// Generate a secure database password - EXACT replica from bitcoin-wallet-ui-iced
fn generate_database_password() -> String {
    use std::collections::hash_map::DefaultHasher;
    use std::hash::{Hash, Hasher};

    let mut hasher = DefaultHasher::new();

    // Use username
    if let Ok(username) = std::env::var("USER") {
        username.hash(&mut hasher);
    } else if let Ok(username) = std::env::var("USERNAME") {
        username.hash(&mut hasher);
    }

    // Use home directory
    if let Some(home) = dirs::home_dir() {
        home.to_string_lossy().hash(&mut hasher);
```

1. **Tracing**: Initialize structured logging with environment-based filter (set `RUST_LOG=debug` for verbose output)

3. **Tauri**: Build the application with plugins, managed state, and all 13 registered commands.

Compare this with the admin app's `main.rs` (Chapter 17A): the admin app does not have phases 1 or 2 — it has no database and no logging. The wallet app's additional complexity is entirely in service of persistence.
