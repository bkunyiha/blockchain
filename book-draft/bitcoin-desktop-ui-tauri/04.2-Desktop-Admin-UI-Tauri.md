# Chapter 4.2: Desktop Admin Interface (Tauri) — Architecture and Code Walkthrough

This chapter explains the `bitcoin-desktop-ui-tauri` crate as a Rust developer reads it: **how a Tauri 2 app bridges a Rust backend to a React/TypeScript frontend**, **how commands flow across the IPC boundary**, and **how the frontend calls our Bitcoin implementation's admin API** through that bridge.

The goal is that you can read this chapter without having the project open:

- **complete Rust backend coverage** lives in Chapter 4.2A (config, models, services, commands)
- **complete React frontend coverage** lives in Chapters 4.2B (infrastructure) and 4.2C (pages)
- each section uses a consistent **Methods involved** box
- diagrams make the IPC boundary and data flow explicit

## What this UI is (in one sentence)

The Tauri desktop admin UI is a **two-language desktop application**: a Rust backend exposes 22 `#[tauri::command]` handlers that wrap our `bitcoin-api` crate, while a React/TypeScript frontend calls those handlers through Tauri's IPC bridge using `invoke()`.

> **Methods involved**
>
> - `main` (Tauri app setup, state management, command registration)
> - `BitcoinApiService::*` (service layer wrapping `AdminClient` HTTP calls)
> - `commands::*` (22 Tauri command handlers across 6 modules)
> - `commands.ts` → `invoke()` (TypeScript-to-Rust bridge)
> - React pages (18 views), components (14 shared), hooks (2 custom)

---

## Why Tauri? (Iced vs. Tauri — an architectural comparison)

In Chapter 4 we built the same admin dashboard with Iced — a pure-Rust, single-language MVU framework. This chapter builds the identical feature set with Tauri, which gives us a **Rust backend paired with a web frontend**. Both approaches have real trade-offs, and we include both in this project so the reader can compare them directly.

| Aspect | Iced (Chapter 4.1) | Tauri (this chapter) |
|--------|-------------------|----------------------|
| **Languages** | Rust only | Rust backend + TypeScript/React frontend |
| **UI paradigm** | MVU (Model-View-Update) | Component-based (React) with IPC bridge |
| **State management** | Single `AdminApp` struct | Zustand (client) + React Query (server) |
| **Async boundary** | Manual Tokio runtime in background thread (`spawn_on_tokio`) | Tauri handles async natively — `#[tauri::command]` functions are async |
| **Event vocabulary** | `Message` enum with ~50 variants | React Router routes + React Query cache keys |
| **Styling** | Iced built-in widgets (programmatic) | Tailwind CSS (declarative) |
| **Shared code** | `bitcoin-api` crate (Rust) | `bitcoin-api` crate (Rust) — same crate, same API calls |
| **Binary size** | Smaller (pure Rust) | Larger (embeds Chromium webview) |
| **Ecosystem** | Iced widgets only | Full npm ecosystem (React Query, Zod, Lucide, etc.) |

The key point: both apps call exactly the same `AdminClient` methods from the `bitcoin-api` crate. The difference is where the UI layer lives (Rust vs. browser) and how events cross the boundary (Iced `Message` vs. Tauri `invoke`).

## Diagram: Tauri IPC architecture (how data flows)

```
┌─────────────────────────────────────────────────────────────────────────┐
│  TAURI APPLICATION                                                      │
│                                                                         │
│  ┌──────────────────────────┐    IPC     ┌────────────────────────────┐ │
│  │   React Frontend         │  invoke()  │   Rust Backend             │ │
│  │   (WebView / Chromium)   │ ────────>  │   (src-tauri/)             │ │
│  │                          │            │                            │ │
│  │  Page Component          │            │  #[tauri::command]         │ │
│  │    └─ useQuery()         │            │    └─ reads State<RwLock>  │ │
│  │       └─ commands.ts     │            │       └─ BitcoinApiService │ │
│  │          └─ invoke(name) │  <──────── │          └─ AdminClient    │ │
│  │             {payload}    │  Result<V> │             └─ HTTP call   │ │
│  └──────────────────────────┘            └─────────────┬──────────────┘ │
│                                                        │                │
└────────────────────────────────────────────────────────│────────────────┘
                                                         │ HTTP
                                                         v
                                              ┌────────────────────┐
                                              │  Bitcoin API Server │
                                              │  (Chapter 3)       │
                                              │  :8080             │
                                              └────────────────────┘
```

Every user action in the browser follows this path:

1. **React page** triggers a query or form submission
2. **`commands.ts`** calls `invoke('command_name', { args })` — this crosses the IPC bridge
3. **Tauri routes** to the matching `#[tauri::command]` Rust function
4. **Command handler** reads `ApiConfig` from shared `RwLock` state
5. **`BitcoinApiService`** creates an `AdminClient` and makes an HTTP request
6. **Result** flows back: `AdminClient` → service → command → IPC → React Query cache → UI re-render

## Mental model (one sentence)

This UI is a **thin Rust command layer** (22 functions that delegate to `AdminClient`) paired with a **React frontend** that uses React Query for caching and Zustand for UI state, with Tauri's IPC bridge (`invoke`) as the only connection between the two worlds.

## Key architectural patterns

### Pattern 1: The Tauri command pattern

Every Rust command follows the same three-step shape:

1. **Read the shared config** from `State<RwLock<ApiConfig>>`
2. **Delegate to `BitcoinApiService`** which creates an `AdminClient` and makes the HTTP call
3. **Return `Result<Value, String>`** — Tauri serializes this as JSON back to the frontend

Here is a representative example — `get_blockchain_info` — showing all three layers:

**Layer 1: Command handler** (`commands/blockchain.rs`)

```rust
use crate::config::ApiConfig;
use crate::services::BitcoinApiService;
use serde_json::Value;
use std::sync::RwLock;
use tauri::State;

#[tauri::command]
pub async fn get_blockchain_info(config: State<'_, RwLock<ApiConfig>>) -> Result<Value, String> {
    let cfg = config.read().map_err(|e| format!("Lock error: {}", e))?;
    BitcoinApiService::get_blockchain_info(&cfg).await
}
```

The `#[tauri::command]` attribute tells Tauri to register this function as a callable command. `State<'_, RwLock<ApiConfig>>` is Tauri's managed state — it injects the shared config automatically. We acquire a read lock (not a write lock, since we only need the URL and API key), then delegate to the service layer.

**Layer 2: Service** (`services/bitcoin_api.rs`)

```rust
pub struct BitcoinApiService;

impl BitcoinApiService {
    pub async fn get_blockchain_info(config: &ApiConfig) -> Result<Value, String> {
        let client = AdminClient::new(&config.base_url, &config.api_key);
        match client.get_blockchain_info().await {
            Ok(response) => {
                if response.success {
                    Ok(json!(response.data))
                } else {
                    Err(response.error.unwrap_or_else(|| "Unknown error".to_string()))
                }
            }
            Err(e) => Err(format!("API error: {}", e)),
        }
    }
}
```

Every service method creates a fresh `AdminClient`, makes the async call, and normalizes the response: extract `data` on success, extract the error message on failure. We create a new client per call (rather than holding a persistent connection) because the user can change `base_url` and `api_key` at runtime through the settings dialog.

**Layer 3: Frontend invocation** (`lib/commands.ts`)

```typescript
import { invoke } from '@tauri-apps/api/core';

export const commands = {
  getBlockchainInfo: () => invoke('get_blockchain_info'),
  // ... 21 more commands
};
```

The frontend calls `invoke('get_blockchain_info')` which crosses the Tauri IPC bridge, runs the Rust function, and returns the result as a JavaScript object. The command name string must exactly match the Rust function name.

### Pattern 2: Shared config with `RwLock`

The app needs a single `ApiConfig` (base URL + API key) that is:

- **Read** by every command handler (22 functions reading concurrently)
- **Written** only by `update_config` (when the user changes settings)
- **Thread-safe** because Tauri runs commands on multiple threads

We solve this with `RwLock<ApiConfig>` managed as Tauri state:

```rust
fn main() {
    let api_config = ApiConfig::default();

    tauri::Builder::default()
        .manage(RwLock::new(api_config))   // Tauri injects this into every command
        // ...
}
```

The `RwLock` allows many concurrent readers but exclusive writers — perfect for a "read-often, write-rarely" config object. See **Chapter 4.2A** for the full `main.rs` walkthrough.

### Pattern 3: React Query for server state

On the frontend, we use TanStack React Query to manage all data from the Rust backend:

```typescript
const { data, isLoading, error } = useQuery({
  queryKey: ['blockchain-info'],
  queryFn: () => commands.getBlockchainInfo(),
  refetchInterval: 30_000,     // Re-poll every 30 seconds
  staleTime: 5 * 60 * 1000,   // Cache valid for 5 minutes
});
```

This gives us automatic caching, background refetching, loading states, and error handling — all without manual state management. Every page component follows this pattern. See **Chapter 4.2B** for the full React Query setup.

## The Iced comparison: what Tauri replaces

If you read Chapter 4, you saw how Iced requires manual wiring for async work:

| Iced pattern | Tauri equivalent |
|-------------|-----------------|
| `init_runtime()` — start Tokio in a background thread | Not needed — Tauri manages its own async runtime |
| `spawn_on_tokio(future)` — bridge sync MVU to async | Not needed — `#[tauri::command] async fn` is natively async |
| `Message::FetchXxx` → `Task::perform(...)` → `Message::XxxLoaded(Result)` | `invoke('command_name')` → `Promise<Result>` — one call, one result |
| `Message` enum (~50 variants for all events) | React Router routes + React Query cache keys (declarative) |
| `view(&AdminApp) -> Element<Message>` (Rust rendering) | JSX components with Tailwind CSS (browser rendering) |
| `AdminApp` struct (single state tree) | Zustand (UI state) + React Query (server state) |

The biggest difference: in Iced, every async operation requires a round-trip through the `Message` enum (request variant → async work → loaded variant → state update → re-render). In Tauri, the frontend calls `invoke()`, gets a Promise, and React Query handles the rest. The "event vocabulary" problem disappears entirely.
