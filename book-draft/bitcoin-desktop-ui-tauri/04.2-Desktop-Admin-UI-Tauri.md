# Chapter 17: Desktop Admin Interface (Tauri) вҖ” Architecture and Code Walkthrough

> **Prerequisites**: This chapter assumes familiarity with the Web API (Chapter 15) and basic React/TypeScript concepts. If you read the Iced admin UI (Chapter 16), you will see the same blockchain features exposed through a different architecture вҖ” but this chapter is self-contained and can be read independently.

**What you will learn in this chapter:** How Tauri bridges a Rust backend to a React/TypeScript frontend in a single desktop application, how the IPC command layer replaces traditional HTTP calls, and how the same `bitcoin-api` crate powers both the Iced and Tauri admin interfaces.

**What is Tauri?** Tauri is a framework for building desktop applications that pair a Rust backend with a web-technology frontend (HTML/CSS/JS). Unlike Electron, Tauri does not bundle Chromium вҖ” it uses the operating system's native webview, resulting in much smaller binaries (typically 5вҖ“24 MB vs. 150+ MB). The Rust side exposes *commands* via `#[tauri::command]`, which the frontend calls through an `invoke()` function over an IPC bridge. The result is a two-language architecture: Rust handles business logic and system access, while React handles the UI.

This chapter explains the `bitcoin-desktop-ui-tauri` crate as a Rust developer reads it: **how a Tauri 2 app bridges a Rust backend to a React/TypeScript frontend**, **how commands flow across the IPC boundary**, and **how the frontend calls our Bitcoin implementation's admin API** through that bridge.

Complete Rust backend coverage lives in Chapter 17A (config, models, services, commands), and the React frontend is covered in Chapters 17B (infrastructure) and 17C (pages).

- diagrams make the IPC boundary and data flow explicit

## What this UI is (in one sentence)

The Tauri desktop admin UI is a **two-language desktop application**: a Rust backend exposes 22 `#[tauri::command]` handlers that wrap our `bitcoin-api` crate, while a React/TypeScript frontend calls those handlers through Tauri's IPC bridge using `invoke()`.

The Tauri desktop admin UI consists of: `main` (Tauri app setup, state management, command registration), `BitcoinApiService::*` (service layer wrapping `AdminClient` HTTP calls), `commands::*` (22 Tauri command handlers across 6 modules), `commands.ts` вҶ’ `invoke()` (TypeScript-to-Rust bridge), and React pages (18 views), components (14 shared), hooks (2 custom).

---

## Why Tauri? (Iced vs. Tauri вҖ” an architectural comparison)

In Chapter 16 we built the same admin dashboard with Iced вҖ” a pure-Rust, single-language MVU framework. Chapter 17 builds the identical **feature set** (blockchain info, wallet management, mining controls, health checks) with Tauri, which gives us a **Rust backend paired with a web frontend**. For detailed feature walkthroughs, see **Chapter 16** вҖ” both UI implementations expose the same admin operations.

This chapter focuses on the **architectural differences** and Tauri-specific patterns (IPC, React Query, Zustand). Both approaches have real trade-offs:

| Aspect | Iced (Chapter 16) | Tauri (this chapter) |
|--------|-------------------|----------------------|
| **Languages** | Rust only | Rust backend + TypeScript/React frontend |
| **UI paradigm** | MVU (Model-View-Update) | Component-based (React) with IPC bridge |
| **State management** | Single `AdminApp` struct | Zustand (client) + React Query (server) |
| **Async boundary** | Manual Tokio runtime in background thread (`spawn_on_tokio`) | Tauri handles async natively вҖ” `#[tauri::command]` functions are async |
| **Event vocabulary** | `Message` enum with ~50 variants | React Router routes + React Query cache keys |
| **Styling** | Iced built-in widgets (programmatic) | Tailwind CSS (declarative) |
| **Shared code** | `bitcoin-api` crate (Rust) | `bitcoin-api` crate (Rust) вҖ” same crate, same API calls |
| **Binary size** | Smaller (pure Rust) | Larger (embeds Chromium webview) |
| **Ecosystem** | Iced widgets only | Full npm ecosystem (React Query, Zod, Lucide, etc.) |

The key point: both apps call exactly the same `AdminClient` methods from the `bitcoin-api` crate. The difference is where the UI layer lives (Rust vs. browser) and how events cross the boundary (Iced `Message` vs. Tauri `invoke`).

Every user action in the browser follows this path:

1. **React page** triggers a query or form submission
2. **`commands.ts`** calls `invoke('command_name', { args })` вҖ” this crosses the IPC bridge
3. **Tauri routes** to the matching `#[tauri::command]` Rust function
4. **Command handler** reads `ApiConfig` from shared `RwLock` state
5. **`BitcoinApiService`** creates an `AdminClient` and makes an HTTP request
6. **Result** flows back: `AdminClient` вҶ’ service вҶ’ command вҶ’ IPC вҶ’ React Query cache вҶ’ UI re-render

## Key architectural patterns

### Pattern 1: The Tauri command pattern

Every Rust command follows the same three-step shape:

1. **Read the shared config** from `State<RwLock<ApiConfig>>`
2. **Delegate to `BitcoinApiService`** which creates an `AdminClient` and makes the HTTP call
3. **Return `Result<Value, String>`** вҖ” Tauri serializes this as JSON back to the frontend

Here is a representative example вҖ” `get_blockchain_info` вҖ” showing all three layers:

**Layer 1: Command handler** (`commands/blockchain.rs`)

```rust
use crate::config::ApiConfig;
use crate::services::BitcoinApiService;
use serde_json::Value;
use std::sync::RwLock;
use tauri::State;

#[tauri::command]
pub async fn get_blockchain_info(
    config: State<'_, RwLock<ApiConfig>>
) -> Result<Value, String> {
    let cfg = config.read().map_err(|e| format!("Lock error: {}", e))?;
    BitcoinApiService::get_blockchain_info(&cfg).await
}
```

The `#[tauri::command]` attribute tells Tauri to register this function as a callable command. `State<'_, RwLock<ApiConfig>>` is Tauri's managed state вҖ” it injects the shared config automatically. We acquire a read lock (not a write lock, since we only need the URL and API key), then delegate to the service layer.

**Layer 2: Service** (`services/bitcoin_api.rs`)

```rust
pub struct BitcoinApiService;

impl BitcoinApiService {
    pub async fn get_blockchain_info(
        config: &ApiConfig
    ) -> Result<Value, String> {
        let client = AdminClient::new(&config.base_url, &config.api_key);
        match client.get_blockchain_info().await {
            Ok(response) => {
                if response.success {
                    Ok(json!(response.data))
                } else {
                    let err_msg = response.error.unwrap_or_else(
                        || "Unknown error".to_string()
                    );
                    Err(err_msg)
                }
            }
            Err(e) => Err(format!("API error: {}", e)),
        }
    }
}
```

Every service method creates a fresh `AdminClient`, makes the async call, and normalizes the response: extract `data` on success, extract the error message on failure. We create a new client per call (rather than holding a persistent connection) because the user can change `base_url` and `api_key` at runtime through the settings dialog.

**Layer 3: Frontend invocation** (`lib/commands.ts`)

```typescript
import { invoke } from '@tauri-apps/api/core';

export const commands = {
  getBlockchainInfo: () => invoke('get_blockchain_info'),
  // ... 21 more commands
};
```

The frontend calls `invoke('get_blockchain_info')` which crosses the Tauri IPC bridge, runs the Rust function, and returns the result as a JavaScript object. The command name string must exactly match the Rust function name.

### Pattern 2: Shared config with `RwLock`

The app needs a single `ApiConfig` (base URL + API key) that is:

- **Read** by every command handler (22 functions reading concurrently)
- **Written** only by `update_config` (when the user changes settings)
- **Thread-safe** because Tauri runs commands on multiple threads

We solve this with `RwLock<ApiConfig>` managed as Tauri state:

```rust
fn main() {
    let api_config = ApiConfig::default();

    tauri::Builder::default()
        // Tauri injects this into every command
        .manage(RwLock::new(api_config))
        // ...
}
```

The `RwLock` allows many concurrent readers but exclusive writers вҖ” perfect for a "read-often, write-rarely" config object. See **Chapter 17A** for the full `main.rs` walkthrough.

### Pattern 3: React Query for server state

On the frontend, we use TanStack React Query to manage all data from the Rust backend:

```typescript
const { data, isLoading, error } = useQuery({
  queryKey: ['blockchain-info'],
  queryFn: () => commands.getBlockchainInfo(),
  refetchInterval: 30_000,     // Re-poll every 30 seconds
  staleTime: 5 * 60 * 1000,   // Cache valid for 5 minutes
});
```

This gives us automatic caching, background refetching, loading states, and error handling вҖ” all without manual state management. Every page component follows this pattern. See **Chapter 17B** for the full React Query setup.

## The Iced comparison: what Tauri replaces

If you read Chapter 4, you saw how Iced requires manual wiring for async work:

| Iced pattern | Tauri equivalent |
|-------------|-----------------|
| `init_runtime()` вҖ” start Tokio in a background thread | Not needed вҖ” Tauri manages its own async runtime |
| `spawn_on_tokio(future)` вҖ” bridge sync MVU to async | Not needed вҖ” `#[tauri::command] async fn` is natively async |
| `Message::FetchXxx` вҶ’ `Task::perform(...)` вҶ’ `Message::XxxLoaded(Result)` | `invoke('command_name')` вҶ’ `Promise<Result>` вҖ” one call, one result |
| `Message` enum (~50 variants for all events) | React Router routes + React Query cache keys (declarative) |
| `view(&AdminApp) -> Element<Message>` (Rust rendering) | JSX components with Tailwind CSS (browser rendering) |
| `AdminApp` struct (single state tree) | Zustand (UI state) + React Query (server state) |

The biggest difference: in Iced, every async operation requires a round-trip through the `Message` enum (request variant вҶ’ async work вҶ’ loaded variant вҶ’ state update вҶ’ re-render). In Tauri, the frontend calls `invoke()`, gets a Promise, and React Query handles the rest. The "event vocabulary" problem disappears entirely.
