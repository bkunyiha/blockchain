# Chapter 17A — Tauri Desktop Admin: Rust Backend

This companion chapter prints every Rust source file in the Tauri desktop admin backend. We walk through the files in dependency order: **dependencies first** (`Cargo.toml`), then **configuration**, **models**, **service layer**, **command handlers**, and finally **app boot** (`main.rs`).

The methods involved in the Tauri backend are: `ApiConfig::default()` and `ApiConfig::new()` (configuration), `BitcoinApiService::*` (25 async methods wrapping `AdminClient`), `commands::blockchain::*`, `commands::wallet::*`, `commands::transactions::*`, `commands::mining::*`, `commands::health::*`, `commands::settings::*` (22 Tauri commands), and `main()` (Tauri builder setup).

## Listing 17A-1 — Dependencies (`Cargo.toml`)

Before we look at any Rust code, we need to understand what crates we depend on. This is the complete `Cargo.toml` for the Tauri backend:

```toml
[package]
name = "bitcoin-desktop-ui-tauri"
version = "0.1.0"
edition = "2021"

[dependencies]
tauri = { version = "2", features = [] }
tauri-plugin-clipboard-manager = "2"
tauri-plugin-shell = "2"
serde = { version = "1", features = ["derive"] }
serde_json = "1.0"
tokio = { version = "1", features = ["rt-multi-thread", "macros"] }
tracing = "0.1"
tracing-subscriber = { version = "0.3", features = ["env-filter", "fmt"] }
bitcoin-api = { path = "../../bitcoin-api", features = ["client", "wallet", "admin"], default-features = false }

[build-dependencies]
tauri-build = { version = "2", features = [] }
```

The key dependencies:

- **`tauri 2`** — The framework itself. Tauri 2 (not 1) provides the command system, window management, plugin architecture, and the IPC bridge to the frontend. We enable no extra features because the defaults give us everything we need.
- **`tauri-plugin-clipboard-manager 2`** — Allows the frontend to read/write the system clipboard (we use this for "copy to clipboard" buttons throughout the UI).
- **`tauri-plugin-shell 2`** — Allows opening external URLs in the system browser.
- **`serde` + `serde_json`** — Serialization. Every value crossing the IPC bridge must be serializable. We derive `Serialize`/`Deserialize` on all our model structs.
- **`tokio`** — Async runtime with multi-threaded support. Tauri uses Tokio internally, and our `#[tauri::command] async fn` handlers run on this runtime.
- **`tracing` + `tracing-subscriber`** — Structured logging. We initialize the subscriber in `main()` and it captures logs from both our code and the `bitcoin-api` crate.
- **`bitcoin-api`** — Our shared crate (path dependency at `../../bitcoin-api`). We enable three features: `client` (the HTTP client), `wallet` (wallet-related types), and `admin` (admin endpoints). This is the same crate used by the Iced desktop app and the web UI — all three frontends share this API client.

## Listing 17A-2 — Configuration (`config/mod.rs`)

The configuration module defines the `ApiConfig` struct that holds the base URL and API key for connecting to our Bitcoin API server. Every command handler reads this config to know where to send HTTP requests.

```rust
use serde::{Deserialize, Serialize};
use std::env;

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct ApiConfig {
    pub base_url: String,
    pub api_key: String,
}

impl Default for ApiConfig {
    fn default() -> Self {
        let base_url = env::var("BITCOIN_API_URL")
            .unwrap_or_else(|_| "http://127.0.0.1:8080".to_string());
        let api_key = env::var("BITCOIN_API_ADMIN_KEY")
            .unwrap_or_else(|_| "admin-secret".to_string());

        ApiConfig { base_url, api_key }
    }
}

impl ApiConfig {
    pub fn new(base_url: String, api_key: String) -> Self {
        ApiConfig { base_url, api_key }
    }
}
```

The `Default` implementation checks for environment variables first (`BITCOIN_API_URL`, `BITCOIN_API_ADMIN_KEY`), falling back to localhost defaults if they are not set. This lets us configure the app for different environments (development vs. production) without recompiling.

We derive `Serialize` and `Deserialize` so the config can be sent to the frontend through the `get_config` command. We derive `Clone` because the `RwLock` read guard gives us a reference, and some call sites need an owned copy.

The `new()` constructor exists for cases where we want to create a config with explicit values — for example, during testing.

## Listing 17A-15 — Data Models (`models/mod.rs`)

These structs define the shape of data that flows between the service layer and the frontend. While the `BitcoinApiService` returns `serde_json::Value` (untyped JSON) for most responses, these models are available for scenarios where we want stronger typing.

```rust
use serde::{Deserialize, Serialize};

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct WalletAddress {
    pub address: String,
    pub label: Option<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct BlockSummary {
    pub hash: String,
    pub height: u64,
    pub timestamp: u64,
    pub transaction_count: u32,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct SendTransactionRequest {
    pub from_address: String,
    pub to_address: String,
    pub amount_satoshis: u64,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct CreateWalletRequest {
    pub label: Option<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct CreateWalletResponse {
    pub address: String,
    pub label: Option<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct WalletInfo {
    pub address: String,
    pub label: Option<String>,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct TransactionInfo {
    pub txid: String,
    pub amount: u64,
    pub timestamp: u64,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct BlockchainInfo {
    pub chain: String,
    pub blocks: u64,
    pub headers: u64,
    pub difficulty: f64,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MempoolInfo {
    pub size: u32,
    pub bytes: u64,
}

#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct MiningInfo {
    pub blocks: u64,
    pub difficulty: f64,
    pub mining_address: Option<String>,
}
```

Notice that every struct derives the same four traits: `Debug` (for logging), `Clone` (for value copying), `Serialize` (Rust → JSON for IPC), and `Deserialize` (JSON → Rust for incoming requests). This is the standard pattern for any data that crosses the Tauri IPC boundary.

The models mirror the shapes returned by the `bitcoin-api` crate's endpoints. `WalletAddress` has an optional `label` because wallets can be created without one. `MiningInfo` has an optional `mining_address` because the node may not have a mining address configured.

**Pattern analysis.** Every method follows the same shape:

1. Create `AdminClient::new(&config.base_url, &config.api_key)`
2. Call the appropriate async method on the client
3. Match on the `Result`:
   - `Ok(response)` → check `response.success` → return `Ok(json!(response.data))` or `Err(response.error)`
   - `Err(e)` → return `Err(format!("API error: {}", e))`

This repetition is intentional — it keeps each method self-contained and independently testable. The `json!()` macro converts the typed `response.data` into a `serde_json::Value`, which is what Tauri serializes back to the frontend. We use untyped JSON here (rather than our model structs) because the frontend can handle arbitrary shapes, and it avoids having to maintain strict type parity between the API server's evolving response format and our local models.

The `send_transaction` method is the most interesting because it constructs a `SendTransactionRequest` from the individual parameters — the frontend sends `from_address`, `to_address`, and `amount_satoshis` as separate arguments, and we assemble them into the request struct that `bitcoin-api` expects. Notice that we call `.to_string()` on the `&str` parameters because `SendTransactionRequest` owns its `String` fields.

## Listing 17A-5 — Command Module Organization (`commands/mod.rs`)

This module file declares the six domain sub-modules and re-exports all 22 command functions so that `main.rs` can reference them as `commands::get_blockchain_info` without needing to know which sub-module they live in.

```rust
pub mod blockchain;
pub mod health;
pub mod mining;
pub mod settings;
pub mod transactions;
pub mod wallet;

pub use blockchain::{get_all_blocks, get_block_by_hash, get_blockchain_info, get_latest_blocks};
pub use health::{health_check, liveness_check, readiness_check};
pub use mining::{generate_blocks, get_mining_info};
pub use settings::{check_connection, get_config, update_config};

    get_address_transactions, get_all_transactions, get_mempool, get_mempool_transaction,

    create_wallet, get_all_addresses, get_balance, get_tx_history, get_wallet_info,
    send_transaction,

```

The `pub use` re-exports are what make the flat `commands::function_name` syntax work in `main.rs`. Without them, we would need to write `commands::blockchain::get_blockchain_info`, which is verbose and couples the command registration to the internal module structure.

## Listing 17A-20 — Blockchain Commands (`commands/blockchain.rs`)

Four async commands for querying blockchain state. Each command acquires a read lock on `ApiConfig` and delegates to `BitcoinApiService`.

```rust
use crate::config::ApiConfig;
use crate::services::BitcoinApiService;
use serde_json::Value;
use std::sync::RwLock;
use tauri::State;

#[tauri::command]
pub async fn get_blockchain_info(config: State<'_, RwLock<ApiConfig>>) -> Result<Value, String> {
    let cfg = config.read().map_err(|e| format!("Lock error: {}", e))?;
    BitcoinApiService::get_blockchain_info(&cfg).await
}

#[tauri::command]
pub async fn get_latest_blocks(config: State<'_, RwLock<ApiConfig>>) -> Result<Value, String> {
    let cfg = config.read().map_err(|e| format!("Lock error: {}", e))?;
    BitcoinApiService::get_latest_blocks(&cfg).await
}

#[tauri::command]
pub async fn get_all_blocks(config: State<'_, RwLock<ApiConfig>>) -> Result<Value, String> {
    let cfg = config.read().map_err(|e| format!("Lock error: {}", e))?;
    BitcoinApiService::get_all_blocks(&cfg).await
}

#[tauri::command]
pub async fn get_block_by_hash(
    hash: String,
    config: State<'_, RwLock<ApiConfig>>,
) -> Result<Value, String> {

}
```

The `#[tauri::command]` attribute is a proc macro that generates the IPC glue code. It handles: deserializing the arguments from JSON, injecting `State<...>` dependencies, serializing the return value back to JSON, and routing errors. The `State<'_, RwLock<ApiConfig>>` parameter is special — Tauri injects it automatically from the state registered in `main.rs` via `.manage(RwLock::new(api_config))`.

Notice `get_block_by_hash` takes a `hash: String` parameter — this comes directly from the frontend's `invoke('get_block_by_hash', { hash: "abc..." })` call. Tauri deserializes the `{ hash }` JSON object into the function's named parameters.

## Listing 17A-11 — Settings Commands (`commands/settings.rs`)

Three commands for runtime configuration management. Unlike all other commands (which are `async`), `get_config` and `update_config` are **synchronous** — they only touch the in-memory `RwLock`, no HTTP calls needed.

```rust
use crate::config::ApiConfig;
use crate::services::BitcoinApiService;
use serde_json::{json, Value};
use std::sync::RwLock;
use tauri::State;

#[tauri::command]
pub fn get_config(config: State<'_, RwLock<ApiConfig>>) -> Result<Value, String> {
    let cfg = config.read().map_err(|e| format!("Lock error: {}", e))?;
    Ok(json!({
        "base_url": cfg.base_url,
        "api_key": cfg.api_key
    }))
}

#[tauri::command]
pub fn update_config(
    base_url: String,
    api_key: String,
    config: State<'_, RwLock<ApiConfig>>,
) -> Result<Value, String> {
    let mut cfg = config.write().map_err(|e| format!("Lock error: {}", e))?;
    cfg.base_url = base_url;
    cfg.api_key = api_key;
    Ok(json!({
        "success": true,
        "base_url": cfg.base_url,
        "api_key": cfg.api_key
    }))
}

#[tauri::command]
pub async fn check_connection(config: State<'_, RwLock<ApiConfig>>) -> Result<bool, String> {
    let cfg = config.read().map_err(|e| format!("Lock error: {}", e))?;
    match BitcoinApiService::health_check(&cfg).await {
        Ok(_) => Ok(true),
        Err(_) => Ok(false),
    }
}
```

**`get_config`** reads the current config and constructs a JSON object manually with `json!({})`. We build the JSON by hand rather than serializing the `ApiConfig` struct because we want to control exactly which fields are exposed to the frontend.

**`update_config`** is the only command that acquires a **write lock** (`.write()` instead of `.read()`). This blocks all concurrent readers until the write is complete. Since updates happen rarely (only when the user changes settings), this is acceptable. After writing, we return the new values as confirmation.

**`check_connection`** is async because it makes an HTTP call. It wraps the health check in a boolean — `true` if the API server responds, `false` if it doesn't. Notice it returns `Result<bool, String>` rather than `Result<Value, String>` — the simpler return type works because Tauri can serialize `bool` just as easily as `Value`.

## Listing 17A-12 — Application Entrypoint (`main.rs`)

This is where everything comes together. The `main` function boots logging, creates the shared config, registers all plugins and commands, and starts the Tauri application.

```rust
mod commands;
mod config;
mod models;
mod services;

use config::ApiConfig;
use std::sync::RwLock;

fn main() {
    // 1) Configure structured logging (controlled by `RUST_LOG`).
    tracing_subscriber::fmt()
        .with_env_filter(tracing_subscriber::EnvFilter::from_default_env())
        .init();

    // 2) Create the shared API configuration.
    //    Reads from BITCOIN_API_URL / BITCOIN_API_ADMIN_KEY env vars,
    //    or falls back to localhost defaults.
    let api_config = ApiConfig::default();

    // 3) Build and run the Tauri application.
    tauri::Builder::default()
        // Wrap the config in RwLock and register it as managed state.
        // Every #[tauri::command] that takes State<RwLock<ApiConfig>>
        // will receive this automatically.
        .manage(RwLock::new(api_config))

        // Register plugins.
        .plugin(tauri_plugin_clipboard_manager::init())
        .plugin(tauri_plugin_shell::init())

        // Register all 22 command handlers.
        // The generate_handler! macro creates the dispatch table that
        // routes invoke("command_name") calls to the correct Rust function.
        .invoke_handler(tauri::generate_handler![
            // Blockchain commands
            commands::get_blockchain_info,
            commands::get_latest_blocks,
            commands::get_all_blocks,
            commands::get_block_by_hash,
            // Wallet commands
            commands::create_wallet,
            commands::get_wallet_info,
            commands::get_balance,
            commands::send_transaction,
            commands::get_tx_history,
            commands::get_all_addresses,
            // Transaction commands
            commands::get_mempool,
            commands::get_mempool_transaction,
            commands::get_all_transactions,
            commands::get_address_transactions,
            // Mining commands
            commands::get_mining_info,
            commands::generate_blocks,
            // Health commands
            commands::health_check,
            commands::liveness_check,
            commands::readiness_check,
            // Settings commands
            commands::get_config,
            commands::update_config,
            commands::check_connection,
        ])
        .run(tauri::generate_context!())
        .expect("error while running tauri application");
}
```

**Step 1: Logging.** We initialize `tracing_subscriber` with an env filter, which means you can control log verbosity at runtime with `RUST_LOG=debug` or `RUST_LOG=bitcoin_desktop_ui_tauri=trace`. The `.with_target(false)` call is omitted here (unlike the Iced app) — Tauri's default output format includes the target module, which is useful for debugging IPC issues.

**Step 2: Config.** `ApiConfig::default()` reads environment variables or falls back to hardcoded defaults. We create it once and wrap it in `RwLock` for thread-safe shared access.

**Step 3: Tauri builder.** The builder pattern chains together:
- `.manage(RwLock::new(api_config))` — registers managed state (dependency injection for commands)
- `.plugin(...)` — registers platform plugins (clipboard, shell)
- `.invoke_handler(tauri::generate_handler![...])` — the dispatch table mapping command name strings to Rust functions
- `.run(tauri::generate_context!())` — starts the application with the configuration from `tauri.conf.json`

**Contrast with Iced's `main.rs`** (Chapter 4.A): The Iced app needs `init_runtime()` to start a background Tokio runtime because Iced's MVU loop is synchronous. Tauri manages its own async runtime internally — our `async fn` commands "just work" without manual runtime setup.

## Listing 17A-13 — Tauri Configuration (`tauri.conf.json`)

This JSON file configures the Tauri application's build pipeline, window properties, security policy, and plugin permissions.

```json
{
  "productName": "Bitcoin Desktop UI",
  "version": "0.1.0",
  "identifier": "com.bitcoin.desktop-ui",
  "build": {
    "frontendDist": "../dist",
    "devUrl": "http://localhost:5173",
    "beforeDevCommand": "npm run dev",
    "beforeBuildCommand": "npm run build"
  },
  "app": {
    "title": "Bitcoin Desktop UI",
    "windows": [
      {
        "title": "Bitcoin Desktop UI",
        "width": 1280,
        "height": 800,
        "resizable": true,
        "fullscreen": false
      }
    ],
    "security": {
      "csp": "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'"
    }
  },
  "plugins": {
    "clipboard-manager": {
      "all": true
    },
    "shell": {
      "open": true
    }
  }
}
```

Key configuration:

- **`build.frontendDist`**: Points to `../dist` — the Vite build output. Tauri bundles this directory into the final application binary.
- **`build.devUrl`**: During development, Tauri loads the frontend from Vite's dev server at `localhost:5173`, which gives us hot module replacement (HMR).
- **`build.beforeDevCommand` / `beforeBuildCommand`**: Tauri automatically runs `npm run dev` (starts Vite) or `npm run build` (production build) before launching.
- **`app.windows`**: We create a single window at 1280×800 pixels, resizable, not fullscreen. This is the main application window.
- **`app.security.csp`**: The Content Security Policy restricts what the embedded webview can load. `'self'` means only resources from the app bundle. `'unsafe-inline'` for styles is required because Tailwind CSS generates inline styles. We do not allow `'unsafe-eval'` — no dynamic script execution.
- **`plugins`**: `clipboard-manager.all: true` grants full clipboard access. `shell.open: true` allows opening URLs in the system browser.
