## Listing 18: `src/main.rs`

This file is the **application entry point**. In the Wallet UI, it is responsible for wiring together three prerequisites before the Iced event loop begins:

- **Async runtime**: the UI performs HTTP requests; those futures must run on Tokio.
- **Encrypted persistence**: settings and wallet addresses are stored in SQLCipher (Chapter 20).
- **MVU wiring**: Iced is launched with the `update` function and the `view` function.

The most important things to understand as you read this file are:

- **Where control transfers to Iced**: the call to `application(...).run_with(...)` is the moment the “traditional main function” ends and the MVU loop begins.
- **What is initialized *before* the UI starts**: `init_runtime()` and `database::init_database(...)` are global preconditions used throughout the program.
- **How the database key is derived**: `generate_database_password()` makes encryption transparent to the user (no password prompt), at the cost of portability (a DB file is not meant to move across machines/users).

> **Methods involved**
> - `main`
> - `generate_database_password`

```rust
mod api;
mod app;
mod database;
mod runtime;
mod types;
mod update;
mod view;

use app::WalletApp;
use iced::{Theme, application};
use runtime::init_runtime;
use update::update;
use view::view;

fn main() -> iced::Result {
    // Initialize Tokio runtime for async operations
    init_runtime();

    // Initialize database with SQLCipher encryption
    // Generate a secure password based on machine/user-specific data
    let db_password = generate_database_password();
    if let Err(e) = database::init_database(&db_password) {
        eprintln!("Failed to initialize database: {}", e);
        // Continue anyway - settings will use defaults
    }

    // Run the application
    application("Bitcoin Wallet UI", update, view)
        .theme(|_| Theme::Dark)
        .run_with(WalletApp::new)
}

/// Generate a secure database password
/// Uses a combination of machine-specific and user-specific data
/// This ensures the database is encrypted but doesn't require user input
fn generate_database_password() -> String {
    use std::collections::hash_map::DefaultHasher;
    use std::hash::{Hash, Hasher};

    let mut hasher = DefaultHasher::new();

    // Use username
    if let Ok(username) = std::env::var("USER") {
        username.hash(&mut hasher);
    } else if let Ok(username) = std::env::var("USERNAME") {
        username.hash(&mut hasher);
    }

    // Use home directory
    if let Some(home) = dirs::home_dir() {
        home.to_string_lossy().hash(&mut hasher);
    }

    // Use application name
    "bitcoin-wallet-ui".hash(&mut hasher);

    // Convert to hex string
    format!("{:x}", hasher.finish())
}
```

## Listing 19: `src/runtime.rs`

This module provides the **Tokio integration layer** for a desktop GUI. The Wallet UI needs Tokio because the HTTP client stack behind the API calls expects it.

The code here answers two questions:

- **Where is the Tokio runtime created and kept alive?**: `init_runtime()` creates a runtime and keeps it running in a background thread.
- **How do we run a future on that runtime from the MVU update loop?**: `spawn_on_tokio()` takes an arbitrary future and ensures it runs on the Tokio runtime, then yields its output back to Iced.

The most important line for the rest of the application is this pattern (see `spawn_on_tokio`):

- `Task::perform(spawn_on_tokio(api_call(...)), Message::SomeCompletion)`

Once you understand that pattern, every async action in `update.rs` becomes straightforward to read.

> **Methods involved**
> - `init_runtime`
> - `spawn_on_tokio`

```rust
use std::sync::OnceLock;

// Store the Tokio runtime handle globally so tasks can access it
static TOKIO_HANDLE: OnceLock<tokio::runtime::Handle> = OnceLock::new();

pub fn init_runtime() {
    // Create a Tokio runtime for async operations
    // This must outlive the application to keep the reactor running
    let rt = tokio::runtime::Runtime::new().expect("Failed to create Tokio runtime");

    // Store the handle globally so it can be accessed from any thread
    TOKIO_HANDLE
        .set(rt.handle().clone())
        .expect("Failed to set Tokio handle");

    // Keep the runtime alive in a background thread
    std::thread::spawn(move || {
        rt.block_on(async {
            // Keep the runtime alive indefinitely
            std::future::pending::<()>().await;
        });
    });
}

// Helper function to wrap a future to ensure it runs on Tokio runtime
pub fn spawn_on_tokio<F>(fut: F) -> impl std::future::Future<Output = F::Output> + Send
where
    F: std::future::Future + Send + 'static,
    F::Output: Send + 'static,
{
    let handle = TOKIO_HANDLE
        .get()
        .expect("Tokio runtime not initialized")
        .clone();
    async move { handle.spawn(fut).await.unwrap() }
}
```

## Listing 5.3: `src/types.rs`

This file defines the **event protocol** of the entire UI:

- `Menu` is the navigation state used by the view to decide which screen to render.
- `Message` is the complete set of events that can happen in the program, including:
  - user input (typing into fields),
  - commands (button presses like `CreateWallet`),
  - async completions (like `WalletCreated(Result<...>)`),
  - and “UI chrome” interactions (submenu hover open/close, editor actions).

When reading the Wallet UI, treat `Message` as the table of contents for behavior:

- If you want to know **what happens on a click**, find the `Message::...` emitted by the view and then locate that match arm in `update.rs`.
- If you want to know **which screens exist**, start with `Menu` and `Menu::ALL`.

Important details to notice:

- `Menu` includes submenu items (`WalletCreate`, `WalletInfo`) even though `Menu::ALL` only contains the top-row buttons.
- Completion messages carry `Result<ApiResponse<T>, String>` so the update loop can handle both network failures and API-level failures (`success == false`).

> **Methods involved**
> - `Menu::submenu_items`
> - `Menu::fmt` (via `Display`)

```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum Menu {
    Wallet,
    WalletCreate,
    WalletInfo,
    GetBalance,
    Send,
    History,

impl Menu {
    pub const ALL: [Menu; 5] = [
        Menu::Wallet,
        Menu::GetBalance,
        Menu::Send,

impl core::fmt::Display for Menu {
```

## Listing 5.4: `src/app.rs`

This module defines the **Model** in MVU: `WalletApp`.

This is what the UI “remembers.” Every screen, widget, and async response is rendered from this one struct.

What to focus on:

- **Startup behavior**: `WalletApp::new()` loads persisted settings and wallet addresses from the encrypted database.
- **Selection as state**: `active_wallet_address` is the single value that gates the rest of the UI (balance/history/send). If it is `None`, those screens are disabled and the update loop will refuse to navigate.
- **Why editor contents are stored**: the various `text_editor::Content` fields exist so displayed values (addresses, txids, JSON) are selectable/copyable and keep their state across re-renders.

If you want to understand “why a screen is disabled,” look at:

- `saved_wallets` and `active_wallet_address` (here),
- and the gating logic in `view.rs` and `update.rs`.

> **Methods involved**
> - `WalletApp::default` (via `Default`)
> - `WalletApp::new`

```rust

                active_wallet_address,
                wallet_info_data: None,
                wallet_balance_data: None,
```

## Listing 5.5: `src/api.rs`

This module contains the **typed async API calls** used by the Wallet UI. These functions do not touch UI state directly; they return `Result<ApiResponse<T>, String>` and let `update.rs` decide how to interpret and display outcomes.

What to focus on:

- **Client construction**: each call creates a `WalletClient` from `ApiConfig` (base URL + API key).
- **Endpoint selection**: each method calls a corresponding `WalletClient` method (create wallet, send transaction, fetch info/balance/history).
- **Error normalization**: errors are converted to `String` so the update loop can display them in `app.status` without needing to depend on error types.

Important for the reader:

- These functions are intentionally thin. The “interesting” logic (validation, UI clearing, caching JSON, status messages, persistence) lives in `update.rs`.

> **Methods involved**
> - `create_wallet`
> - `send_tx`
> - `fetch_wallet_info`
> - `fetch_balance`
> - `fetch_address_transactions`


## Listing 5.6: `src/update.rs`

This module is the **core behavioral engine** of the Wallet UI. In MVU terms, it is the Update function: it takes a `Message`, mutates `WalletApp`, and returns a `Task<Message>` for any async work that should run in the background.

How to read this file effectively:

- **Read it as a state machine**: each `Message::Variant` is one transition.
- **Identify invariants**:
  - wallet-dependent screens and actions require `active_wallet_address`,
  - navigation away from Send clears send-form state to avoid stale UX.
- **Track the “async boundary”**: look for `Task::perform(...)` and `Task::batch(...)`. Those are the only places the update loop initiates HTTP work.
- **Find completion handling**: any message ending in `...Loaded`, plus `WalletCreated` and `TxSent`, is where results are interpreted and editors/JSON caches are populated.

The most important parts for understanding the UI are:

- `Message::MenuChanged` (navigation + auto-fetch),
- `Message::SelectWallet` (selection + parallel fetch fan-out),
- `Message::CreateWallet`/`WalletCreated` (remote create + local persistence + UI refresh),
- `Message::SendTx`/`TxSent` (send + txid display),
- `Message::CopyToClipboard`/`ClipboardCopied` (platform clipboard integration).

> **Methods involved**
> - `update`

```rust
use crate::api;
use crate::app::WalletApp;
use crate::runtime::spawn_on_tokio;
use crate::types::{Menu, Message};
use bitcoin_api::{ApiConfig, CreateWalletRequest};
use iced::Task;

pub fn update(app: &mut WalletApp, message: Message) -> Task<Message> {
    match message {
        Message::MenuChanged(m) => {
            // Clear Send Bitcoin data when navigating away from Send menu
            if app.menu == Menu::Send && m != Menu::Send {
                app.to.clear();
                app.amount.clear();
                app.last_txid = None;
                app.transaction_id_editor = iced::widget::text_editor::Content::new();
            }

            // Set menu and close submenu when selecting any menu item
            // Check if menu requires active wallet
            let requires_wallet = matches!(m, Menu::GetBalance | Menu::Send | Menu::History);
            if requires_wallet && app.active_wallet_address.is_none() {
                // Don't allow navigation to these menus without an active wallet
                app.status = "Please select a wallet first".into();
            } else if m == Menu::WalletInfo && app.saved_wallets.is_empty() {
                // Don't allow navigation to Wallet Info without any wallets
                app.status = "No wallets available. Please create a wallet first.".into();
            } else {
                app.menu = m;
                app.wallet_submenu_open = false; // Close submenu when selecting any menu item

                // Auto-fetch data when navigating to these sections if wallet is selected
                if let Some(address) = &app.active_wallet_address {
                    let cfg = ApiConfig {
                        base_url: app.base_url.clone(),
                        api_key: Some(app.api_key.clone()),
                    };
                    let address_clone = address.clone();

                    match m {
                        Menu::WalletInfo => {
                            // Clear previous data and fetch wallet info
                            app.wallet_info_data = None;
                            app.wallet_info_editor = iced::widget::text_editor::Content::new();
                            return Task::perform(
                                spawn_on_tokio(api::fetch_wallet_info(cfg, address_clone)),
                                Message::WalletInfoLoaded,
                            );
                        }
                        Menu::GetBalance => {
                            // Clear previous data and fetch balance
                            app.wallet_balance_data = None;
                            app.wallet_balance_editor = iced::widget::text_editor::Content::new();
                            return Task::perform(
                                spawn_on_tokio(api::fetch_balance(cfg, address_clone)),
                                Message::BalanceLoaded,
                            );
                        }
                        Menu::History => {
                            // Clear previous data and fetch transaction history
                            app.transaction_history_data = None;
                            app.transaction_history_editor =
                                iced::widget::text_editor::Content::new();
                            return Task::perform(
                                spawn_on_tokio(api::fetch_address_transactions(cfg, address_clone)),
                                Message::TransactionHistoryLoaded,
                            );
                        }
                        _ => {}
                    }
                }
            }
```

## Listing 5.7: `src/view.rs`

This module is the **rendering layer** of the Wallet UI. It is pure: it reads `&WalletApp` and produces `Element<Message>` values that describe what should be drawn and which `Message` to emit when the user interacts.

How to read this file:

- Start at `view(app)`: it composes the top toolbar, navigation, the active section, and the status bar.
- Then read the section helpers:
  - `wallet_list_section`: displays wallets from `saved_wallets` and emits `SelectWallet`.
  - `create_wallet_section`: emits `CreateWallet` and displays `new_address` via a selectable editor.
  - `wallet_info_section`, `get_balance_section`, `history_section`: display cached JSON editors that are filled by async completions.
  - `send_section`: emits `SendTx` and displays `last_txid` via a selectable editor.
  - `settings_section`: edits base URL/API key and emits `SaveSettings`.

Important patterns to notice:

- **UI gating**: buttons/menus are disabled when there is no active wallet. This is UX; the update loop also enforces correctness.
- **Selectable displays**: long addresses/JSON are shown using `text_editor` + `scrollable`, which requires editor state stored in the model.
- **Submenu hover**: the Wallet submenu is implemented using `mouse_area` and `app.wallet_submenu_open`.

If you are unsure “where a click goes,” find the `.on_press(Message::...)` in the view and then locate that message in `update.rs`.
