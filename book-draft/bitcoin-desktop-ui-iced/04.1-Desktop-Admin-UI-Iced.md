# Chapter 25: Desktop Admin Interface (Iced) — Architecture and Code Walkthrough

> **Prerequisites:**: This chapter assumes you have read the Web API chapter (Chapter 24) — the desktop UI is an HTTP client to that API. No prior GUI framework experience is needed; we introduce Iced from scratch below. Familiarity with Rust enums and pattern matching is essential, as the entire UI is driven by message dispatch.

> If you need a refresher on enums
> and pattern matching, see Chapter 33 (Rust Language Guide, Sections on Enums and Pattern Matching).

> **What you will learn in this chapter:**
> - Build a desktop admin interface using the Iced framework's Model-View-Update pattern
> - Implement state management, async API calls, and UI event handling in pure Rust
> - Design modular UI components with styling and layout
> - Integrate with the bitcoin-api crate for blockchain communication

> **Framework Comparison:** This chapter builds the admin interface in pure Rust using Iced's Model-View-Update pattern. Chapter 26 builds the same interface using Tauri 2 (Rust backend + React frontend). Reading both chapters gives you a direct comparison of two architectures solving the same problem — one single-language, one polyglot.

> **Scope:** This chapter builds an admin interface for monitoring and interacting with the blockchain. We do not cover accessibility (a11y) compliance, internationalization (i18n), or platform-specific packaging (DMG, MSI, AppImage).

**What is Iced?** Iced is a cross-platform GUI framework for Rust inspired by Elm. It uses a Model-View-Update architecture: your application state lives in a single struct (the *model*), user interactions produce *messages* (a Rust enum), and a pure `update` function handles each message by returning the new state. A separate `view` function renders the current state into a widget tree. If you have used Elm, Redux, or SwiftUI, the pattern will be familiar — but Iced is pure Rust with no JavaScript or web runtime involved.

> **Note:** The Model-View-Update pattern used here is inspired by Elm. If you are familiar with Elm, Redux, or other unidirectional data flow architectures, Iced's approach will feel immediately familiar.

This chapter explains the `bitcoin-desktop-ui` crate as a Rust developer reads it: **how an Iced MVU app boots**, **how messages flow through the update loop**, and **how the UI triggers async HTTP calls** to our Rust Bitcoin implementation’s admin API.

Complete method coverage lives in the companion walkthrough chapters (16A–16C).

## What this UI is (in one sentence)

The desktop admin UI is a **thin MVU shell**: it renders state (`view.rs`), mutates state in response to messages (`update.rs`), and performs async admin calls (`api.rs`) on a Tokio runtime (`runtime.rs`).

The methods involved in this architecture are: `main` (boot + theme + app start), `init_runtime` and `spawn_on_tokio` (async boundary), `AdminApp::{new, clear_related_data}` (state and screen cleanup), `update` (message dispatcher), and `view` and `view_*` helpers (rendering).

---

## Diagram: MVU event loop (Iced)

```text
User input (click / type)
   |
   v
Message (enum variant)  -----+
   |                         |
   v                         |
update(&mut AdminApp, Message)
   |
   |-- pure state change -> Task::none()
   |-- async request -> Task::perform() -+
   |                                      |
   v                                      |
view(&AdminApp) -> Element<Message>       |
                                   (later) |
                                Message::XxxLoaded(Result)
```

---

## Diagram: async boundary (HTTP requests)

```text
update (sync)        runtime (tokio)           api (async)
─────────────        ───────────────           ───────────
Task::perform(   spawn_on_tokio(fut)  ---->  AdminClient::new()
  spawn_on_tokio,     .await                .endpoint().await
  Message::Loaded                          .send()
)
```

## Exercises

1. **Add a Refresh Button** — Add a "Refresh" button to the admin UI that reloads blockchain data from the API. Define a new `Message::Refresh` variant, implement the `Update` handler to trigger an API call, and add the button to the `View`. Observe how the MVU cycle processes this new interaction.

2. **MVU Cycle Trace** — Pick any user interaction (e.g., clicking "Mine Block") and trace the complete MVU cycle: which `Message` variant is generated, what the `Update` function does, what side effects (API calls, state changes) occur, and how the `View` reflects the new state.
